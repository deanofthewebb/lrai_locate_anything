# Single-file TRT/ONNX deployment design ŌĆö LocateAnything-3B

_Generated by workflow wes8qeg23 + digest ww1cbyxgo on 2026-06-02._

## Headline recommendation

**Option B: Two engines for production** ŌĆö `vision_proj.engine` (MoonViT + patch-merge + projector fused) and `llm.engine` (single LLM engine combining prefill + AR decode via `IIfConditional` on `past_seq_len == 0`, dropping MTP/SDLM from the prod path). Host drives the AR loop.

Single-engine fusion is rejected because TRT 10.15.1 forbids `IRecurrenceLayer` inputs with changing shapes (hard blocker on growing KV cache inside `ILoop`), and no published precedent exists for an in-graph AR loop; even TRT-LLM keeps the loop host-side. Two files is the canonical NVIDIA multimodal pattern (encoder + LLM) and gives ~2.0 GB + ~1.8 GB Colab footprint vs the current 5-engine ~4.5 GB sprawl.

## Current state (5-engine sprawl)

| # | Engine | Role | Notes |
|---|--------|------|-------|
| 1 | `vision.engine` | MoonViT-SO-400M encoder | fp16, fixed-resolution (`pos_emb_baked`, `min=opt=max` profile) |
| 2 | `projector.engine` | MLP projector | fp16; fed by `python_patch_merger` running in numpy at `orchestrator.py:191-197` |
| 3 | `llm_prefill.engine` | LLM prefill | STRONGLY_TYPED bf16, bf16 KV I/O, fp32 logits |
| 4 | `llm_decode_MTP.engine` | LLM decode (SDLM/MTP block-mask) | bf16; ~6.36 GB; K hard-capped at 6 in optimization profile |
| 5 | `llm_decode_AR.engine` | LLM decode (AR fallback) | bf16; ~6.36 GB; differs from MTP engine by only 147,456 B (constant-folded mask topology) |

**Totals:** ~19.87 GiB on disk; three LLM engines are nearly identical builds. Collapsing the three LLM engines to one saves ~12.7 GB. `n_layers = 36` (NOT 72 ŌĆö confirmed by polygraphy dumps ending at `present_v_35`).

**Cross-engine precision boundaries** carry an implicit fp16<->bf16 reinterpret via `uint16` byte-storage proxies in `engine.py`.

I/O contract for the unified LLM ONNX:

- Inputs: `input_ids` int64 `(1, S_or_K)`, `visual_features` bf16 `(L_post, 2048)`, `position_ids` int64 `(1, S_or_K)`, `attention_mask` int64 `(1, total_len)`, `past_k_i` / `past_v_i` bf16 `(1, 2, P, 128)` for `i in 0..35`. `P = 0` at prefill, `P > 0` at decode.
- Outputs: `logits` float32 `(1, S_or_K, 152681)`, `present_k_i` / `present_v_i` bf16 `(1, 2, P + S_or_K, 128)`.
- Branch true (prefill): ignores past tensors, scatters `visual_features` at image-token positions.
- Branch false (decode): concatenates `past_kv` with the new step.
- TRT 10 should deduplicate identical constants across the two `IIfConditional` branches (verify with `engine.inspector`).

## Why not other options

- **Option A ŌĆö single `.engine` with in-graph `ILoop` AR decode**: TRT 10.15.1 forbids `IRecurrenceLayer` inputs from changing shape across iterations, so a growing KV cache is illegal inside `ILoop`. Pre-allocating max KV is legal but triggers the "memory allocated as if all dynamic dimensions take maximum value" clause, blowing Colab VRAM. Zero published precedent ŌĆö TRT-LLM itself keeps the AR loop host-side.
- **Option C ŌĆö TRT-LLM as the runtime**: v0.19.0 Qwen2.5-VL support is PyTorch-backend only, NOT `trtllm-build`. MoonViT-SO-400M has no recipe; SDLM/PBD has no analog (MTP support is DeepSeek-V3 specific). Adopting TRT-LLM means ~200 LOC `vision_runner.py`, a custom `build_multimodal_engine.py` extension, and a Triton-style 6-submodel repo ŌĆö strictly more files than today.
- **Option D ŌĆö keep 5 engines, ship together**: propagates the `python_patch_merger` CPU hop into Colab and keeps the MTP/AR split that exists only for SDLM (an R&D speed feature, not a correctness feature).
- **Option E ŌĆö fuse only vision+projector, leave 3 LLM engines**: saves the CPU patch-merger hop but still ships three LLM engines. The `IIfConditional` prefill/decode merge is the bigger artifact-count win.
- **Option F ŌĆö drop bf16, use fp16-everywhere to widen TRT-LLM path**: regression from commit `7fd1449`'s validated bf16 STRONGLY_TYPED configuration with no offsetting benefit.

## Why TRT-LLM is not the answer

TRT-LLM v0.19.0 added Qwen2.5-VL only via the PyTorch backend (`Qwen2_5_VLForConditionalGeneration`); the TensorRT engine-compile path (`trtllm-build`) still lists only Qwen2-VL as recipe-supported, and the HuggingFace card for LocateAnything-3B explicitly disclaims TRT-LLM support. MoonViT-SO-400M has no `build_multimodal_engine.py` recipe (Qwen2.5-VL's official tower is the standard Qwen ViT, which LocateAnything swaps), and SDLM/PBD has no analog because TRT-LLM's MTP support is hard-wired for DeepSeek-V3/R1 module shape. Citations: `https://nvidia.github.io/TensorRT-LLM/release-notes.html`, `https://github.com/NVIDIA/TensorRT-LLM/blob/main/examples/models/core/multimodal/README.md`, `https://huggingface.co/nvidia/LocateAnything-3B/blob/main/README.md`.

## Why a single-file ONNX with Loop is not the answer

An ONNX `Loop`+`If` model is theoretically expressible at the spec level but practically broken: the TensorRT EP cannot consume it reliably (falls back to CPU/CUDA and kills the TRT speedup), the CUDA EP works but loses `past_present_share_buffer` reuse, and there is no public production example of a VLM shipped as a single `.onnx` with an in-graph AR loop. The canonical onnxruntime-genai pattern is multi-file: `genai_config.json` names separate `decoder.filename`, optional `encoder.filename`, `embedding.filename`, `vision.filename`, plus a `pipeline[].filename` list, with the AR loop driven externally by the C++ runtime (`generate()` / `generate_next_token()`). The closest mainstream single-file precedent ŌĆö Optimum's `decoder_model_merged.onnx` ŌĆö only merges prefill + decode-with-past under an `If` node; the AR loop is still external. Citations: `https://onnxruntime.ai/docs/genai/reference/config.html`, `https://huggingface.co/docs/optimum/en/exporters/onnx/usage_guides/export_a_model`, `https://onnx.ai/onnx/operators/onnx__Loop.html`.

## TRT 10 control-flow analysis

| Mechanism | Status | Notes |
|-----------|--------|-------|
| `IIfConditional` | **VIABLE** | Supports runtime-computed 0-D bool predicates (e.g. `past_kv.shape[seq] == 0`). As of TRT 10.11, per-dimension shapes may differ between branches (rank+dtype must still match). Execution is lazy ŌĆö only the selected branch's layers run. STRONGLY_TYPED does NOT block it. Quantitative overhead is undocumented. |
| `ILoop` (true growing KV cache) | **BLOCKED** | TRT 10.15.1 release notes: "Inputs to the `IRecurrenceLayer` must always have the same shape. ONNX models with loops whose recurrence inputs change shapes will be rejected." Second blocker: loops with multiple dynamic dimensions allocate memory assuming ALL dims simultaneously hit max ŌĆö pathological for AR decode. |
| `PluginV3`-owned AR loop | **NO PRECEDENT** | Newly viable (PluginV2 is forbidden inside `ILoop`/`IIfConditional`; PluginV3 is the first generation supported in control-flow scopes). Every published NVIDIA plugin example is single-shot. Not realistic without significant R&D. |

**Published single-engine precedents for AR LM with in-graph generation loop: none.** TRT-LLM ŌĆö the canonical AR-LM-on-TRT implementation ŌĆö deliberately uses a host-side `PyExecutor` loop that re-invokes `ModelEngine` per token. Single-engine prefill+decode in TRT-LLM is achieved by one optimization profile spanning both regimes plus a host phase flag, NOT by `IIfConditional` or `ILoop`. Citations: `https://docs.nvidia.com/deeplearning/tensorrt/latest/inference-library/work-with-conditionals.html`, `https://docs.nvidia.com/deeplearning/tensorrt/latest/inference-library/work-with-loops.html`, `https://docs.nvidia.com/deeplearning/tensorrt/latest/getting-started/release-notes-10/10.15.1.html`.

## Implementation phases

1. **Phase 1: Static patch-merger ONNX fusion** ŌĆö _small / low risk_
   - **Deliverable**: new `export/vision_proj.py` exporting MoonViT + static patch-merger + projector as ONE ONNX graph. Verify numerical parity vs current `vision.engine -> python_patch_merger -> projector.engine` pipeline on 5 sample images (max abs diff < 1e-3 in fp16).
   - **Risk**: patch-merger body is a pure reshape+transpose+reshape (`parse.py:43-61`) and `grid_hws` is baked as a constant buffer (`vision.py:48-52`). Only risk is opset 17 not handling the specific permute pattern cleanly ŌĆö fallback is explicit `Reshape`/`Transpose` ops via `onnx-graphsurgeon`.

2. **Phase 2: Unified LLM ONNX with If-merged prefill/decode_ar** ŌĆö _medium / medium risk_
   - **Deliverable**: new `export/llm_unified.py` producing `llm_unified.onnx` via either (a) `torch.cond` around the `past_k_0.shape[2] == 0` predicate or (b) `onnx-graphsurgeon` merging two existing ONNX files under an If node (Optimum `decoder_model_merged.onnx` pattern). Smoke-test in onnxruntime CUDA EP at P=0 and P=100.
   - **Risk**: `torch.cond` + `dynamo=False` is mutually exclusive (`dynamo=False` is forced in `export_with_external_data` `llm.py:98` to avoid the SDLM `.item()` trace-time crash). Either (a) use `dynamo=True` now that MTP is dropped from this path (no `.item()` problem), OR (b) use the `onnx-graphsurgeon` merge route which sidesteps torch entirely. **Choose (b) as primary, (a) as backup.**

3. **Phase 3: TRT build with `IIfConditional` + strongly-typed bf16** ŌĆö _medium / medium-high risk_
   - **Deliverable**: new `trt/build_unified.py` that builds `llm.engine` from `llm_unified.onnx` with one optimization profile (S `[16,1024,4096]`, K=1, P `[0,1024,4096]`). Verify TRT importer preserves the `If` as `IIfConditionalLayer` (use `trtexec --verbose` or `onnx-graphsurgeon` inspect). Verify weight deduplication via `engine.inspector` ŌĆö expected size ~1.8 GB, not ~3.6 GB.
   - **Risk**: TRT 10's ONNX-If ŌåÆ `IIfConditional` importer has known shape-inference quirks; per-dimension shape mismatch between branches needs TRT 10.11+. Fallback: pad prefill `present_kv` output to `(P_max+S)` with sentinel zeros and mask in the host. **Verify Colab TRT version >= 10.11.**

4. **Phase 4: Host AR-decode runner** ŌĆö _medium / low-medium risk_
   - **Deliverable**: new `trt/runner_unified.py` with a `generate(image, prompt, max_new_tokens=150)` entry. One TRT context, one set of pre-allocated KV buffers (sized to `p_max=4096`, `head_dim=128`, 36 layers, `n_kv_heads` from config), no Python patch merger. First call sets P=0 (prefill branch); subsequent calls set `P=current_len` (decode branch). Stop on `<|im_end|>`. Box-parsing via existing `parse.py`.
   - **Risk**: KV buffer plumbing ŌĆö must use `IExecutionContext.set_tensor_address` with the SAME underlying memory for `past_k_i` in step N+1 as `present_k_i` in step N. Verify with a small reference-counting test.

5. **Phase 5: Colab packaging + benchmark** ŌĆö _small / low risk_
   - **Deliverable**: public Colab notebook that downloads `vision_proj.engine` (~1.9 GB) and `llm.engine` (~1.8 GB) from S3, runs imageŌåÆboxes in <1 minute on T4/L4. Bench against the current 5-engine pipeline for parity (mAP within 0.5pp on the v1.8 sentinel-rays validation set if applicable, else a small visual regression set).
   - **Risk**: Colab T4 may not ship TRT 10.11+. Verify before committing; if older, fall back to two separate LLM engines (`prefill.engine` + `decode_ar.engine`, 3 files total).

_(Optional Phase 6: preserve the R&D 5-engine path under a separate entry point ŌĆö e.g. `lrai_locate_anything.export.build_rd` vs new `lrai_locate_anything.export.build_prod` ŌĆö documented as R&D = 5 engines with MTP/SDLM speedup; prod = 2 engines AR-only.)_

## ONNX equivalent

- **Feasible**: yes.
- **Recommended target**: onnxruntime-genai with CUDA EP (primary); plain onnxruntime CUDA EP with a thin Python AR loop (fallback for environments without onnxruntime-genai). TensorRT EP is NOT recommended for the LLM file because the `IIfConditional` + dynamic KV combo hits the documented control-flow fallback-to-CPU problem.
- **Approach**: three ONNX files following the onnxruntime-genai canonical multimodal split ŌĆö `vision_proj.onnx` (MoonViT + static patch-merger + projector, ~1.9 GB external-data), `embedding.onnx` (Qwen2 `embed_tokens` with image-token scatter of `visual_features` into `inputs_embeds`, ~600 MB), and `model.onnx` (LLM body + `lm_head` with the same If-merged prefill/decode_ar pattern, ~1.8 GB external-data). A `genai_config.json` describes the pipeline and `past_present_share_buffer: true` enables in-place KV across `Run()` calls. The AR loop is driven by onnxruntime-genai's C++ `generate()`/`generate_next_token()`, NOT by an in-graph ONNX `Loop`.
- **Caveats**:
  1. Three files, not one ŌĆö matches NVIDIA/HuggingFace's prevailing pattern but doesn't beat the TRT plan on file count.
  2. onnxruntime-genai requires a model-builder PR or hand-authored `genai_config.json` to recognize MoonViT (Qwen2.5-VL recipe uses standard Qwen ViT, not MoonViT) ŌĆö small but real glue work.
  3. SDLM/MTP is dropped on this path too (no genai analog for block-mask attention).
  4. Single-monolith-with-`Loop` is feasible in spec but every realistic runtime backend will partition or refuse it; do not pursue.

## Open questions for the user

- Is 2 files (`vision_proj.engine` + `llm.engine`) acceptable as "production single artifact", or does the goal require exactly 1? If exactly 1, the only path is concatenating engines into a tar/zip "bundle" ŌĆö TRT itself cannot fuse a vision encoder and an LLM with a host-driven AR loop into one engine handle.
- Is dropping MTP/SDLM from the production path acceptable? It removes the parallel-box-decoding speedup (block of 6 tokens per LLM step) but keeps correctness. The R&D 5-engine path retains MTP. If MTP must stay in prod, file count rises to 3 (`vision_proj` + `llm_prefill_decode_ar` + `llm_decode_mtp`) because the constant-folded SDLM mask branch is incompatible with the If-merged pattern.
- What is the Colab TRT version target? `IIfConditional` with per-dim shape mismatch between branches requires TRT 10.11+. If Colab is pinned older, the merge falls back to two LLM engines (`prefill.engine` + `decode_ar.engine`), giving 3 files total.
- Should the new prod export route live inside `lrai_locate_anything` as a new submodule (e.g. `export_prod/`, `trt_prod/`) or replace the existing `export/` + `trt/`? Constraint says "production single-file should be a SEPARATE export route" ŌĆö read as new submodule, please confirm.
- Bench target on Colab: is L4 the assumed GPU, or must T4 work too? T4 has 16 GB VRAM; bf16 strongly-typed LLM engine is ~1.8 GB plus KV at P=4096 is ~1.1 GB ŌĆö fits L4 (24 GB) comfortably, tight on T4 (16 GB) once vision + activations are added. Need a `max_seq_len` target to size the KV profile.
- Do we want fp4/int4 weight-only quantization on the LLM engine for Colab? `int4.py` exists in `export/` ŌĆö applying it cuts `llm.engine` from ~1.8 GB to ~700 MB, making the total artifact ~2.6 GB (single sub-minute S3 download on Colab). Decide before Phase 3.

## Surprises / caveats worth knowing

- `n_layers` is 36, not 72 ŌĆö the "72 KV pairs" figure conflates layers with K/V tensors. Confirmed by polygraphy outputs ending at `present_v_35`.
- The difference between `llm_decode.engine` (MTP) and `llm_decode_ar.engine` is ONLY a constant-folded attention-mask topology ŌĆö both share the same wrapper and helper; the SDLM `.item()` branch is evaluated at trace time under `dynamo=False` and baked into the graph. Byte difference is 147,456 B out of 6.82 GB.
- TRT-LLM canonically produces TWO engines per VLM, not one ŌĆö the "single artifact" goal cannot be met via TRT-LLM. The TRT-LLM `PyExecutor` itself runs the AR loop host-side; no NVIDIA-validated in-graph AR-LM loop exists.
- TRT 10.15.1 explicitly forbids `IRecurrenceLayer` inputs from changing shape across iterations ŌĆö HARD blocker for any in-graph growing-KV-cache design. Pre-allocating max inside `ILoop` is legal but triggers worst-case dynamic-dim memory allocation, blowing Colab VRAM.
- The three LLM engines total ~19.07 GiB on disk despite being near-identical builds; collapsing to one LLM engine saves ~12.7 GB.
- Vision engine is hard-fixed-resolution by design (`pos_emb_baked` at construction, `min=opt=max` profile). Any unified prod engine inherits this constraint unless MoonViT `pos_emb` is restructured to evaluate at runtime.
